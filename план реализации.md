# План реализации testovoe3 Telegram Rewrite App

## Summary
Собираем с нуля full-stack приложение в `/Users/asmtv1/Desktop/testovoe3`: React + TypeScript, FastAPI, Postgres, Telethon, DeepSeek API. На `testovoe3.gafus.ru` приложение будет доступно через Caddy + HTTPS на сервере `89.125.250.84`. Внутри приложения будет простая авторизация на двух пользователей: `user1 / 12345` для проверяющих и `user2 / 2407041` для тестирования.

Проверенные нюансы: DeepSeek использует OpenAI-compatible API с `base_url=https://api.deepseek.com`, модель берём `deepseek-v4-flash`, thinking mode отключаем через `extra_body={"thinking":{"type":"disabled"}}`; Telethon поддерживает login flow, чтение истории через `iter_messages`, публикацию через `send_message`; Caddy умеет automatic HTTPS при корректном DNS и открытых 80/443. Источники: [DeepSeek models](https://api-docs.deepseek.com/quick_start/pricing), [DeepSeek thinking mode](https://api-docs.deepseek.com/guides/thinking_mode), [Telethon client reference](https://docs.telethon.dev/en/stable/quick-references/client-reference.html), [Telethon signing in](https://docs.telethon.dev/en/stable/basic/signing-in.html), [Caddy reverse proxy](https://caddyserver.com/docs/quick-starts/reverse-proxy).

## Key Changes
- Frontend: Vite + React + TypeScript, один аккуратный интерфейс: app-login, Telegram login wizard, workspace для каналов, постов, prompt, rewrite result и publish.
- Backend: FastAPI API под `/api/*`, cookie-session авторизация, Telethon per-user sessions, DeepSeek rewrite service, Postgres persistence.
- DB: строки постов изолированы по `user_id`; `user1` и `user2` не видят данные друг друга.
- Telegram sessions: отдельные файлы `/app/sessions/user1.session` и `/app/sessions/user2.session`.
- Секреты: DeepSeek key, session secret, encryption key и app users только через `.env`; ключи не попадут во frontend и git.
- Изображения и видео не поддерживаем: берём только текстовые сообщения.

## Implementation Steps
1. **Scaffold**
   - Создать структуру: `backend/`, `frontend/`, `deploy/`, `docker-compose.yml`, `.env.example`, `.gitignore`, `README.md`.
   - Backend dependencies: `fastapi`, `uvicorn`, `sqlalchemy[asyncio]`, `asyncpg`, `pydantic-settings`, `telethon`, `openai`, `cryptography`, `pytest`, `httpx`.
   - Frontend dependencies: `react`, `react-dom`, `vite`, `typescript`.

2. **Docker + Deploy Base**
   - `docker-compose.yml`: `postgres`, `backend`, `frontend`, `caddy`.
   - Caddyfile: `testovoe3.gafus.ru { handle /api/* { reverse_proxy backend:8000 } handle { reverse_proxy frontend:80 } }`.
   - Volumes: Postgres data, Caddy data/config, Telethon sessions.
   - Server prerequisites: DNS A record `testovoe3.gafus.ru -> 89.125.250.84`, open ports `80/tcp`, `443/tcp`.

3. **Backend Config**
   - `.env` keys: `DATABASE_URL`, `SESSION_SECRET`, `APP_USERS=user1:12345,user2:2407041`, `DEEPSEEK_API_KEY`, `DEEPSEEK_BASE_URL=https://api.deepseek.com`, `DEEPSEEK_MODEL=deepseek-v4-flash`, `APP_ENCRYPTION_KEY`.
   - Add `/api/health` returning DB connectivity and app status.
   - Use same-domain cookies, no CORS needed in prod; Vite dev proxy handles local `/api`.

4. **Database Schema**
   - `posts`: `id`, `user_id`, `source_channel`, `target_channel`, `telegram_message_id`, `original_text`, `rewritten_text`, `publish_status`, `error_message`, `created_at`, `updated_at`.
   - Unique index: `(user_id, source_channel, telegram_message_id)`.
   - `telegram_credentials`: `user_id`, `api_id`, encrypted `api_hash`, `phone`, timestamps.
   - `telegram_login_states`: `user_id`, `phone`, encrypted `api_hash`, `api_id`, `phone_code_hash`, `expires_at`.
   - Для тестового допустимо `metadata.create_all()` на старте вместо Alembic.

5. **App Auth**
   - API: `POST /api/auth/login`, `POST /api/auth/logout`, `GET /api/auth/me`.
   - Проверять `APP_USERS` constant-time compare.
   - После login класть `user_id` в signed HttpOnly cookie через Starlette SessionMiddleware.
   - Все Telegram/posts/rewrite/publish endpoints требуют текущего пользователя.

6. **Telegram Login Wizard API**
   - `GET /api/telegram/status`: возвращает `connected`, `phone`, `needs_credentials`.
   - `POST /api/telegram/send-code`: принимает `api_id`, `api_hash`, `phone`; вызывает Telethon `send_code_request`; сохраняет `phone_code_hash` в login state.
   - `POST /api/telegram/sign-in`: принимает `code`; если Telethon вернул `SessionPasswordNeededError`, вернуть `password_required`.
   - `POST /api/telegram/password`: принимает 2FA password; после успеха сохраняет credentials, удаляет login state.
   - `POST /api/telegram/logout`: удаляет Telethon session-файл и credentials текущего app user.

7. **Telegram Client Service**
   - Создавать клиента только для текущего `user_id`.
   - Session path: `/app/sessions/{user_id}.session`.
   - На fetch/publish проверять, что session authorized; если нет, вернуть понятную ошибку `telegram_not_connected`.
   - Обрабатывать `FloodWaitError`, `ChannelPrivateError`, `ChatAdminRequiredError`, неверный channel username/id.

8. **Posts Fetch + Pagination**
   - API: `GET /api/posts?source_channel=@channel&target_channel=@target&offset_id=...`.
   - Page size фиксированный: 10 текстовых постов.
   - Использовать `iter_messages(source, offset_id=offset_id)`.
   - Игнорировать сообщения без текста; медиа не скачивать.
   - Чтобы не объявить конец из-за пачки медиа, сканировать до 100 raw messages за запрос, пока не набрали 10 текстовых или не закончилась история.
   - Response: `items`, `next_offset_id`, `has_more`, `message`.
   - Upsert each text post into `posts` for current `user_id`.

9. **DeepSeek Rewrite**
   - API: `POST /api/posts/{id}/rewrite` body: `{ "prompt": "..." }`.
   - Проверить, что post принадлежит текущему user.
   - Запрос: system message задаёт роль рерайтера, user message содержит prompt и original text.
   - Использовать OpenAI Python SDK with DeepSeek base URL, `model=deepseek-v4-flash`, `extra_body={"thinking":{"type":"disabled"}}`.
   - Сохранять `rewritten_text`, status `rewritten`; при ошибке status `error` + `error_message`.

10. **Publish**
   - API: `POST /api/posts/{id}/publish` body: `{ "target_channel": "...", "text": "..." }`.
   - Публиковать через тот же Telethon user session: `send_message(target_channel, text)`.
   - Перед публикацией требовать непустой текст; сохранять manual edits из textarea.
   - Успех: status `published`; ошибка прав/канала/rate limit: status `error` с читаемым сообщением.

11. **Frontend UI**
   - Первый экран: login form для `user1/user2`.
   - После app-login: Telegram wizard с шагами `credentials + phone -> code -> optional 2FA password -> connected`.
   - Workspace: `source channel`, `target channel`, `Load posts`, список постов, кнопка `Загрузить ещё 10`, prompt textarea, `Rewrite`, editable result textarea, `Publish`.
   - Все операции показывают `loading / success / error`; ошибки выводить рядом с действием.
   - Не делать маркетинговую landing page; первый экран после auth должен быть рабочим интерфейсом.

12. **README**
   - Описать локальный запуск: заполнить `.env`, `docker compose up --build`.
   - Описать prod deploy на `89.125.250.84`: DNS, ports, `.env`, `docker compose up -d --build`.
   - Описать ограничения: только текст, пользователь должен состоять в source channel, в target channel нужны права публикации, Telegram session file чувствителен.
   - Добавить тестовые app-логины без DeepSeek key.

## Assumptions
- Используем app-level auth, не Caddy Basic Auth.
- `user1/12345` и `user2/2407041` нужны именно для входа в приложение; Telegram-аккаунты они подключают отдельно через UI.
- API ID/API hash сохраняем зашифрованными в Postgres, потому что Telethon требует их для повторного открытия session после рестарта backend.
- Telegram phone/code/2FA password не сохраняем.
- DeepSeek API key хранится только в `.env`; уже отправленный в чат ключ лучше перевыпустить после тестирования.
- Один backend instance, без горизонтального масштабирования; для тестового это достаточно.
